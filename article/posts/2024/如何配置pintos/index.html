如何配置pintos

Oct 24, 2024

因为学校的操作系统课程设计要用这玩意，所以就要配置这个玩意
（看之后能说的东西多不多，多的话单开一个分类）
配置这玩意相对还是快的，但我也走了点弯路
我的环境是WSL的Debian

<0x00> 前置

需要配置WSL，然后安装Debian
然后需要git和稳定的维图诺·普莱维特·奈特沃克

<0x01> 拉取pintos仓库

git clone git://pintos-os.org/pintos-anon --depth=1

需要提前配置下git的proxy，不然可能会很慢

<0x02> 配置Debian需要的软件包

# 编译环境
sudo apt install make build-essential
# qemu模拟器
sudo apt install qemu-system

<0x03> 简单配置

修改./src/threads/Make.vars中的SIMULATOR

SIMULATOR = --qemu

新版本Linux不包含stropts.h，我们创建一个空的就可以了

sudo nano /usr/include/stropts.h

然后修改./src/utils/squish-pty.c，注释下面的代码，大概在288行

// 注释这些代码
if (isastream (slave))
{
  if (ioctl (slave, I_PUSH, "ptem") < 0
      || ioctl (slave, I_PUSH, "ldterm") < 0)
    fail_io ("ioctl");
}

这样就可以了
（网上说的其他修改貌似都不是很关键）

<0x04> 尝试运行

在工作路径./src下面一条条执行

# 编译utils的文件
cd ./utils
make
# 回到./src
cd ..
# 编译内核
cd ./threads
make
# 进入编译好的内核文件夹
cd ./build
# 通过提供的脚本运行pintos
../../utils/pintos -- run alarm-multiple

理论上pintos就会运行了

不要用Arch编译这个东西

如果使用Arch编译这个东西的话会发现loader.bin的大小有128M
而不是正常的512字节
不只是WSL中Arch会出现这个问题，物理机Manjaro也会这样
具体原因不是很清楚，我怀疑是gcc版本问题
我测试过Ubuntu22的gcc11和Debian12的gcc12，这两个都可以成功编译
但Arch的是gcc14，可能较高的gcc版本导致编译行为不符合预期
不过我暂时还没尝试在Arch中使用gcc11编译过
（源码安装gcc11，编译就要几个小时，先摸了）
